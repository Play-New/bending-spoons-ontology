# CLAUDE.md — how to work in this repo

## What this is
A rigorous, public **world model + ontology of Bending Spoons**, built **only** from official sources (rule 8) — the SEC filings (Form F-1 `bsp-f1`, final 424B4 prospectus `bsp-424b4`) and Bending Spoons' own published documents (e.g. `sources/hiring/`), all captured in `sources/`. The repo holds **one model and one analysis**: **the model at the root** (`ontology.md` + `world-model/` + `capabilities/` + `interfaces/` — Bending Spoons today, every node `status: confirmed`, except the declared gap) and **`bending-spoons-as-an-intelligence.md`** (the value-migration thesis, `status: proposed` — deliberately an essay, never a second model; the generic move it instantiates is `foundations/org-as-an-intelligence.md`). The **will** and the **sources** serve both. The two contracts these rules rest on are in `foundations/`.

It is a world model you can query + capabilities as invocable skills — not a written analysis. An MCP server (`mcp/`) serves the object nodes through read-only query tools (`get_node`, `list_nodes`, `search`, …) and the capabilities as tools (query · two-phase engine · radar · hire · audit — 27 in all).

## The modeling discipline — governing rules (follow exactly)
The model has **two layers** (there is no third):
- **Semantic** = **objects + properties + links** — *what exists*. Lives in `world-model/`.
- **Kinetic** = **actions + functions + dynamic security** — *what the machine does / what is computed / what gates it*. Lives in `capabilities/`, gated per `foundations/ontology.md §6`.
- **Governance** is **not** a third layer — it is dynamic security over the kinetic layer: the will's **constraints** gate the actions (`will.md`); the will's *intent* moves Bending Spoons.

1. **World-model = objects, only objects.** Every object *type* in the ontology has a node; nothing that isn't an object is a node. Backing datasets (`*.csv`) are labelled *backing*, not object-nodes. The one deliberate exception is `cross-product-graph` (`[gap]`: the missing unified Customer — the thesis hinge).
2. **Property vs function.** A **property** is a value *stored on that object*. A value *computed across objects, or not attributable to a single object,* is a **function / derived property** (lives in `capabilities/functions/`). So `Business.revenue` and `Deal.consideration` are properties; company-level financials and ratios (net income, revenue-per-Spooner) are **functions** over `financials.csv`.
3. **Links connect two object types only**, declared **once** (no inverse duplication), and **never** point to an action or a function.
4. **1:1.** A node's — and a backing dataset's — fields are *exactly* the object's ontology properties. No more, no less.
5. **Deal ⊥ Business ⊥ Interface.** The `Deal` (the event: consideration · enterprise value · deal-type · IRR · cohort · date · PPA) is separate from the `Business` it produces (operating data: revenue · MAU · paying · NRR · tenure · ARPU · channels · margin · status · lifecycle → `businesses.csv`), which is separate from the `Interface`(s) it ships (the app the user touches: brand · category · capabilities-delivered · AI features · live/sunset → `interfaces.csv`; linked `Interface —of→ Business`).
6. **All Target analysis** lives in the customer world model (the `Target` node + `targets.csv`), not in the scouting capability. `Target` properties = the `screen-target` gates, 1:1, + the pipeline-state properties the actions write (status · irr · walk-away price).
7. **Actions** = single transactions that create / modify / delete objects, set properties, or write links (may be function-backed, with write-back). **Functions** = computed readings, no write-back. *(Actions + functions are the ontology's **kinetic layer** — what the `capabilities/` folder and `list_capabilities` hold; a **capability** (the composable, invocable competency, `foundations/org-as-an-intelligence.md §3`) is what they realize. Never equate 'capability' with 'action': see `foundations/org-as-an-intelligence.md §3`.)*
8. **Grounding.** Every fact cites an **official source** with a `~L` (or section) ref against a capture in `sources/`. Two tiers of official source, in order of authority: (1) the **SEC filings** `bsp-f1` / `bsp-424b4` — the primary record for everything about the company as a public filer; (2) **Bending Spoons' own published documents** (e.g. the hiring docs in `sources/hiring/`, cited by id like `bsp-selection-process`) — company-authored, primary for what only they document (their selection process), captured as facts + short fair-use quotes, never verbatim dumps (they are copyrighted). **Third-party press is NOT a fact source** — it is always `[to-validate — press only]`. A third, weaker provenance is used for a few interface brand/feature facts: `provenance: official-site` (a live company page, cited by URL) — honestly tagged and usable, but NOT line-addressable or captured in `sources/`, so it is the least authoritative tier and never carries a model-critical figure. Mark `[derived]`, `[to-validate — press only]`, `[thesis]`, `[decision]`. Model nodes are `status: confirmed`; the analysis is `status: proposed`.
9. **No framework or author names in the files.** The ontology vocabulary (object / property / link / action / function / interface) is used unattributed. Influences are named only in `README.md` → References **and in the two `foundations/` contracts**, which are grounded in and cite their sources by design (that is what a contract is). Everywhere else — the model, the analysis, the capabilities, the interfaces — the vocabulary is used unattributed; `mcp/audit.py` enforces this.
10. **The decision test.** An object earns a node **only if a modeled decision touches it** (a verb writes it · a reading reads it · a link carries it · a will constraint hooks onto it). Otherwise: property, row, or prose (`foundations/ontology.md §4.0`).
11. **The two clocks, on every node.** `as_of` = the reporting period the data reflects; `last_synced` = when the node was last verified against `sources/`. Every edit that changes facts updates `last_synced`; `as_of` advances only with a newer source period. `mcp/audit.py` fails without them — run it after every batch of edits.
