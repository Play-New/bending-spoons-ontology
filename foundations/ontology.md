# The ontology contract

How this repo models a subject as an **ontology** — the grammar every node and capability obeys. It is grounded in the **Palantir Foundry ontology**: the vocabulary is used exactly as Palantir defines it (sources at the end). `CLAUDE.md` is the short enforced version of these rules; this is the full reference.

## 1. What an ontology is
An ontology is the **digital twin** of a subject — *"a digital twin of the organization, containing both the semantic elements (objects, properties, links) and kinetic elements (actions, functions, dynamic security)."* It is *"designed to represent the complex, interconnected **decisions** of an enterprise, **not simply the data**"*: it sits **above** the raw data and gives one coherent, queryable model of what exists and what is done.

The relational analogy is exact and load-bearing:
> **object type = a dataset (schema) · object = a row · property = a column · property value = a field · link type = a JOIN.**

## 2. The two layers
- **Semantic — *what exists*:** **object types · properties · links.**
- **Kinetic — *what is done / what is computed*:** **actions · functions · dynamic security.**

*"The data objects, or 'nouns', … must be complemented by 'verbs' in order to model decisions; semantics must be paired with kinetics."* The whole is a **four-fold integration of data, logic, action, and security** — in this repo: data = the world model, logic = the functions, action = the actions, security = the will's constraints.

**Dynamic security** is the third *kinetic* element: access control evaluated at request time (*who may commit which action, under what state*) — enforcement, never strategy. The security **mechanism** and the **constraints** it enforces are not the same thing, and do not live in the same place. The mechanism lives in the ontology: the `governance` field of each contract (submission criteria + the human gates), enforced by the engine (§6) — subject-agnostic machinery. The **constraints** live in `will.md`, which stays **strategic and brief**: it states what Bending Spoons refuses and requires — never sell, the hurdles, the flat org, founder control — as *policy*, not as code. Only the **mechanically checkable subset** of those refusals is *instantiated* into the security mechanism — the hurdles → submission criteria, the human go/no-go → permissions; the rest (never sell — strategy not law, the charter permits it; give-back; patience; no bid escalation) stays in the will, **honored by the human, gated by nothing**. And the will's **problem → mission → intent** has no security counterpart at all — that is what *moves* the subject (the org contract, `org-as-an-intelligence.md`). So the will sits above both contracts as a short strategy document; the ontology holds the enforcement; each takes its piece. How security is instantiated here is §6.

## 3. The primitives — definition → our operating rule

**Object type** — *"the schema definition of a real-world entity or event."* Backed by data. Our gloss (not Palantir wording): an object type is an entity definition, not a behavior — **distinct from actions** (operations on objects) and from workflows. An **object** is one instance; an **object set** is *"a collection of multiple object instances"* — which we use in two modes: **static** (a fixed list) or **dynamic** (a saved filter that updates as data matches).
→ **Rule:** the world model holds **objects, and only objects** (entities/events). Every object *type* has a node. Collections (e.g. the Portfolio, the Market) are **dynamic object sets**, not types.

**Property** — *"the schema definition of a characteristic"* of the entity; a value **stored on the object**. Carries a **base type**: `string · integer · double · boolean · date · timestamp · geopoint · geoshape · vector · attachment · time-series · media-reference · cipher-text · struct`, and arrays of these (*"excluding the Vector and Time series types"*).
→ **Rule:** a property is a value **stored on that object**, with a declared base type. Per-object financials (`Business.revenue`, `Deal.consideration`) are properties.

**Shared property** — a property usable on **multiple object types** (consistent modeling, centralized metadata).
→ **Rule:** use it for attributes common to several types. Ours: `status` · `as_of` (the reporting period the node's data reflects) · `last_synced` (when the node was last verified against `sources/`) — the freshness pair every node carries.

**Derived property** — *"calculated at runtime based on the values of other properties or links on objects"* (aggregating or selecting over linked objects); **not stored**; carries the **cumulative security** of everything it reads.
→ **Rule:** any value **computed across objects, or a ratio / aggregate not attributable to a single object, is a derived property (a reading) — never a stored property.** Revenue-per-Spooner, the consolidated financials, the portfolio-churn reading are derived. They live in the kinetic layer (`capabilities/functions/`).

**Link type** — *"the schema definition of a relationship between two object types,"* with a **cardinality** — we notate it `1:1 · 1:N · N:N` (our shorthand; the docs name e.g. *many-to-many*); a **link** is one instance. Use-rule: *properties describe an entity; links relate two entities.* Self-referential links (a type to itself) are allowed.
→ **Rule:** a link connects **two object types**, is declared **once** (never both inverse directions), and **never** points to an action, a function, or a non-object. A `target` that resolves to nothing is a defect, not a link. **The `relations:` field in a node's frontmatter is *not* a link** — it is pipeline metadata (hand-offs between capabilities, what a function reads or is about); ontology links are declared only in the subject's `ontology.md §1`, between object types. A frontmatter relation must still resolve — to a node `id`, or to a §1 **type name** when it holds at type level (e.g. `about: product`) — and make sense, but it never enters the semantic layer.

**Action type** — *"the definition of a set of changes or edits to objects, property values, and links that a user can take at once. It also includes the **side effect behaviors** that occur with action submission."* One **transaction**: *"a single transaction that changes the properties of one or more objects, based on a **user-defined logic**"* — and that logic *"could be a simple business rule, a conventional machine learning model, an LLM-driven function, or a complex multi-step orchestration."* Its full anatomy: **parameters** (standardized input) · **logic** (what computes the edits; **function-backed** when a function does) · the **edits** (create / modify / delete objects, set property values, write links) · **submission criteria** (when it may be taken) · **permissions/validations** (who may take it) · **side effects** (notifications, webhooks — behaviors, not edits). It commits by **write-back**: the edits are *"captured in an object type's writeback dataset."*
→ **Rule:** an action **writes** the model, and is the *only* thing that does. The state-changing verbs (source, screen, underwrite, close, transform…) are actions. An *agent* runtime is a kind of **logic** ("a complex multi-step orchestration"), never a different kind of transaction: judgment decides *what to propose*; the commit is always the declared transaction, behind its gate. The full enforced shape is §5.

**Function** — *"a piece of code-based logic that takes in input parameters and returns an output"* — a callable unit. (A *derived property* is function-logic integrated into an object query; a *function* is a standalone unit.)
→ **Rule:** a function is a **read-only computation** returning a reading (monitor, audit, the headline reading). No write-back.

**Interface** — *"an Ontology type that describes the shape of an object type and its capabilities,"* enabling **polymorphism**: many object types implement one interface, so a workflow can act on any of them.
→ **Rule:** use a polymorphic interface only where many object types genuinely share one shape — **none is declared in this model today**. *Every "interface" in this repo (`interfaces/` — the products as surfaces — and the analysis's interchangeable surfaces + `meta-interface`) is the **Dorsey** concept — a delivery surface / the edge where people act — not Palantir polymorphism (see `org-as-an-intelligence.md` §5).*

**Object view** — *"a central hub for all information and workflows related to a particular object."* → in this repo, the object's node file.

**Dynamic security** — access control evaluated at request time. **Mandatory controls** *"propagate along with each unit of data … via provenance and lineage"* (markings for data needing special protection; Organizations as silos); **discretionary controls** are *"roles with different operations (for example, view or edit)"* granted per resource; plus *"granular row or column-level controls based on a user's attributes."* Two tiers in practice: **schema** (who may edit the types) and **data** (attribute-based, per object / property); actions and functions respect it, and a derived reading carries the cumulative security of its inputs.
→ **Rule:** security here = **integrity**, on three tiers (§6): only the human amends the types; the human gates commit the actions (with the will's constraints as submission criteria); provenance and markers travel with every fact and bind its use. The subject's refusals live in `will.md`.

## 4. The modeling rules (all derive from §3)
0. **The decision test — what earns a node.** Model *backwards from decisions*, never forward from inventory: an object type enters §1 **only if a modeled decision touches it** — an action writes it, a function reads it, a link carries it, or a will constraint hooks onto it. Everything else is a property value, a backing row, or prose — *until a decision needs it*. (The ontology represents decisions, not simply data; "does X deserve a node?" always reduces to "which verb writes it, which reading reads it, which constraint hangs on it?")
1. **Model the world, not the source data.** Object types are real-world entities/events — never a copy of the source's tables or an export's columns.
2. **Objects only** in the world model. Backing datasets (`*.csv`) are labelled *backing*, not object-nodes.
3. **Property vs derived property** = *stored-on-the-object* vs *computed-over-objects*. Per-object financials are properties; ratios / consolidated / cross-object aggregates are derived (functions).
4. **Links** connect two object types, declared once, never to a non-object.
5. **1:1** — a node's, and a backing dataset's, fields are **exactly** the object type's properties — plus, in a backing dataset only: the **provenance columns** (`source`, `note` — the §6 mandatory control travelling with each row) and a **foreign-key column per declared §1 link** (link = JOIN; e.g. `deals.csv.produces_business` materializes `Deal —produces→ Business`). Nothing else.
6. **Actions write · functions read · governance gates** (the will's constraints → `will.md`; the security tiers → §6).
7. Every object type has a **stable primary key** (`id`), a **title**, and **typed properties**; every fact is **grounded** in a cited source (`~L` into `sources/`). Naming: a **type node** (one file for the type, instances in its backing csv or aggregate) is **singular**, matching the §1 type name (`deal.md`, `facility.md`, `platform.md`). The two **people-collective** types are named as the collective — `founders.md` and `spooners.md` — a single node that *is* the group, so its name is the group (plural), not the member. A **folder of instance nodes** is **plural** (`businesses/`, `interfaces/`, `tools/`).
8. **The two clocks.** Every node carries the freshness pair (§3, shared properties): **`as_of`** — the reporting period its data reflects — and **`last_synced`** — when it was last verified against `sources/`. They answer different questions (*how old is the fact* vs *how fresh is the verification*) and move independently: an edit that changes a node's facts must update `last_synced`; `as_of` advances only when data from a newer source period enters. A node without the pair, or an edit that leaves `last_synced` stale, is a defect. The verifiable substrate for the pair is the file history (git) — self-declared dates are checked for presence by the audit, for honesty by the log.

## 5. The capability contract — the enforced shape

Every node in `capabilities/` carries a **contract block**. Its fields are not stylistic: each one is an element of the §3 anatomy, so a missing or misplaced field is a modeling defect, not a formatting choice.

**An action contract — the verb + seven fields:**

| field | what it holds | the §3 element it instantiates |
|---|---|---|
| `action` | the ontology verb, declared once in the subject's `ontology.md §2` | the action type's name |
| `parameters` | the typed inputs the user/agent supplies | parameters — standardized input |
| `logic` | what computes the edits: a fixed business rule (skill), a judgment loop (agent), or a function (→ *function-backed*) | the user-defined logic |
| `transaction` | **the single atomic commit** — create / modify / delete objects · set property values · write links. Nothing else in the node changes the world model | the edits, "taken at once" |
| `submission criteria` | when the action may be taken at all — the state gates and aborts | submission criteria |
| `governance` | who may commit — the human gates — plus the will constraints that bind it | permissions/validations + the will's constraints (`will.md`, §6 tier 2) |
| `side-effects` | behaviors fired on submission that are **not** edits: notifications, hand-offs to the next capability, learning loops | side effects |
| `backing` | the **write-back dataset(s)** that capture the committed edits, 1:1 with the object's properties, + the runtime that hosts the logic | the writeback dataset |

**A function contract — three fields, and no more:** `input` (typed parameters) · `computation` (the logic, over which objects/backing it reads) · `output` (the typed reading returned). *"A function takes in input parameters and returns an output"* — it has **no transaction, no side-effects, no write-back**; it carries the cumulative security of everything it reads (§3, derived property).

**The invariants (what the audit service checks — contract in `../mcp/audit-contract.md`, runnable as `../mcp/audit.py`; every violation found by hand becomes a new check):**
1. The contract's verb appears in the subject's `ontology.md §2` **once**, with the **same transaction** — contract and index may not drift. The check runs both ways: every §2 row names its backing contract file, or is explicitly marked *contract pending* — and a pending verb **commits nothing**. A contract covers one verb unless it explicitly declares the others it also backs (as `transform` does for run-experiment and reorganize); the audit checks each covered verb separately.
2. Every property the transaction writes is **declared on the object type in §1** and is a **column of its backing dataset** (the 1:1 rule — which admits only two non-property columns: provenance and link foreign-keys, §4.5). A write to an undeclared property is a defect. The same 1:1 holds **statically**: a node's declared properties are exactly §1's for its type — a property appearing first in a node's frontmatter or Properties block is a schema amendment smuggled as a node edit (§6 tier 1).
3. Every link the transaction writes is **declared once in §1**, object↔object, in the declared direction.
4. Only an action has a `transaction`; a function contract has none. A function that writes is a defect.
5. A side-effect never edits the model — an edit hiding in `side-effects` belongs in the `transaction`. A side-effect that names a state change must name the **action** that commits it (a hand-off names the receiving verb; "X improves/accrues" with no committing action is an illegal edit in disguise).
6. Every gate, rule, and figure in a contract cites its source (`~L` into `sources/`, `will.md` for a constraint, or a `[decision]` marker for a modeling choice).
7. The runtime (skill/agent) may change only the `logic`, never the `transaction` or its gates.
8. **Data tier (§6):** a fact enters the model only with its provenance attached (`~L` into `sources/`); an inference only with its marker (`[derived]` / `[thesis]` / `[to-validate — press only]`). An unsourced fact, or a `[thesis]`/press-only value grounding a `confirmed` property, is a defect. **Markers propagate by weakest-wins**: a reading inherits the weakest marker among its inputs (`confirmed` > `[derived]` > `[thesis]` / `[to-validate — press only]`) — a press-only input can never launder into an unmarked derived figure.
9. **Provenance of edit (§6 tier 2):** every backing-dataset row, and every property value on a node, is attributable to a declared action's transaction. A hand-added row or value with no action that could have written it is corruption, even if its citation resolves.
10. **Function shape:** a `type: function` node carries exactly the three contract fields (`input · computation · output`) and no write verb; an action-shaped block (`does:` with edits, `transaction:`, gates that commit) on a function node is a defect regardless of what it is named.
11. **Freshness (§4.8):** every node carries `as_of` + `last_synced`; an edit that changes facts updates `last_synced`, and `as_of` advances with the source period. A missing pair or a stale-dated edit is a defect.

## 6. The security model — dynamic security, instantiated

This repo needs no access control in the Foundry sense: every fact is public by construction (the SEC filings and Bending Spoons' own published documents), there is one operator, every node is `visibility: shared`. Importing markings, roles, and Organizations would be machinery without a threat model. What **"secure" means here is integrity** — the model cannot be corrupted — and that is dynamic security instantiated on the three tiers this system actually has:

1. **Schema tier — who may edit the types.** Only the human amends the ontology: `ontology.md §1/§2`, the contracts in `foundations/`, and `will.md`. Capabilities (and any agent) *propose*; a schema change is never a side-effect of running one.
2. **Action tier — who may commit, and when.** Every write goes through a declared action, behind its human gate (permissions) and the will's constraints (submission criteria). Nothing else writes: not functions, not side-effects, not agents acting on their own judgment (§5, invariants 4–7).
3. **Data tier — what may enter, and how it may be used.** The mandatory control that travels with each unit of data is **provenance**: the `~L` citation into `sources/` — and provenance has a **time dimension**, the two clocks (§4.8): `as_of` dates the fact, `last_synced` dates its verification. The **markers** are the markings: `[derived]`, `[thesis]`, `[decision]`, `[to-validate — press only]`, and `status: confirmed / proposed` travel with the fact and bind its downstream use — press is never a fact source, a `[thesis]` never grounds a `confirmed` property, and a reading carries the cumulative epistemic status of everything it reads — weakest marker wins (invariant 8; the derived-property security rule, §3, applied to epistemics).

Enforcement is the audit service (§5 invariants, `../mcp/audit.py`); the MCP exposes resources read-only and tools that propose behind the same gates. One boundary case: in the analysis (`../bending-spoons-as-an-intelligence.md §6`), security also becomes modeled **content** — per-purpose consent on the unified graph is genuine attribute-based data-tier security, treated there as the composition primitive.
